Adarsh AI Persona - Detailed Evals Report

This evaluation focuses on the three criteria that matter most for an interview-facing AI persona: voice quality, grounded answer behavior, and booking reliability. I used deterministic prompts, manual inspection, and LLM-as-a-judge scoring to measure not just fluency, but whether the system stayed faithful to real resume and project evidence while preserving a valid scheduling path.

Voice Eval Checked latency, factual accuracy, and task completion in Vapi browser tests.
Chat Eval Checked hallucination rate, retrieval quality, and answer faithfulness.
Booking Eval Checked whether the user always reached a real live booking path with confirmation.

Voice Quality

  • Latency: measured from caller utterance end to first assistant reply in Vapi browser tests, targeting sub-2-second first-response behavior so the interaction still feels live.
  • Accuracy: compared spoken answers against grounded facts for identity, role fit, RAG work, and project tradeoffs.
  • Task Completion: counted a session as successful when the assistant introduced itself correctly, answered the main question, handled at least one follow-up, and preserved a valid interview-booking path.

I tested prompts like "Who are you?", "Why is Adarsh a strong fit for this role?", "Tell me about his RAG experience", and "Can you help schedule an interview?" A good result required both conversational quality and factual discipline.

Chat Groundedness

  • Hallucination Rate: counted answers that introduced unsupported employers, dates, achievements, project claims, or technical details.
  • Retrieval Quality: checked whether top retrieved chunks matched the actual user intent for role-fit, RAG-specific, and GitHub project prompts.
  • Answer Faithfulness: checked whether the final answer stayed aligned with retrieved evidence instead of drifting into plausible but unsupported claims.

This mattered especially for a resume agent, where a confident but unsupported answer is worse than a short honest answer. I therefore treated "sounds good" and "is grounded" as separate evaluation axes.

LLM-as-a-Judge

I used LLM-as-a-judge as a structured evaluator over (question, retrieved evidence, answer). The judge scored groundedness, retrieval relevance, helpfulness, and hallucination risk. It was used as a rubric-based grader, but manual review remained the final authority.

{ "groundedness_score": 1, "retrieval_relevance_score": 1, "helpfulness_score": 1, "hallucination": false, "notes": "" }
  • Is the answer fully supported by the retrieved evidence?
  • Are the retrieved chunks relevant to the real question?
  • Does the answer stay specific without inventing facts?
  • Does the answer remain useful under a follow-up?

Failure Modes Found And Fixed

1. Booking reliability looked stronger than it really was

Problem: Google Calendar and Cal.com integrations were inconsistent across local and deployed environments, which meant the UI could look booking-ready while the provider layer still failed.

Fix: Moved production booking to a stable Calendly handoff so every user-facing booking action resolves to a live scheduling flow with confirmation.

2. Voice scheduling responses sounded unnatural

Problem: The assistant initially tried to read a raw booking URL aloud, which made the experience sound brittle on a live call.

Fix: Rewrote the voice prompt to use a spoken booking path and shorter scheduling language.

3. Public UI exposed weak and low-signal states

Problem: Empty sections, technical-looking panels, and visible retrieval metadata distracted from the actual demo path.

Fix: Simplified the homepage to the three core actions: ask, book, and call, while keeping the grounding layer intact underneath.

What I Would Improve With 2 More Weeks

  1. Add true end-to-end voice booking with tool-driven calendar actions instead of a booking-link handoff.
  2. Build a batch eval harness that runs saved prompts through retrieval, generation, and LLM-as-a-judge scoring to produce reproducible regression metrics.
  3. Expand GitHub indexing so repo-specific answers surface stronger stack, purpose, and tradeoff detail directly from project artifacts.
