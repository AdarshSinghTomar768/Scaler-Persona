Adarsh AI Persona Evals Report

This report evaluates the system as an interview-facing assistant rather than a generic chatbot. The key focus areas were voice quality, grounded chat behavior, and booking reliability under real demo conditions.

1. Evaluation Goal

I wanted to verify three things: whether the voice agent felt live enough to hold a conversation, whether the chat assistant stayed grounded to real resume and project evidence, and whether the booking path remained reliable even when lower-level provider integrations were unstable. The evaluation therefore mixed deterministic prompts, manual review, and LLM-as-a-judge scoring rather than relying only on subjective spot checks.

2. Voice Quality

  • Latency: measured from caller utterance end to first assistant response in Vapi browser tests, targeting sub-2-second first reply behavior.
  • Accuracy: compared spoken answers against grounded facts for identity, role fit, RAG work, and project tradeoffs.
  • Task completion: a session counted as successful when the assistant introduced itself correctly, handled at least one follow-up, and preserved a valid booking path.

I used short scripted calls such as "Who are you?", "Why is Adarsh a strong fit for this role?", "Tell me about his RAG experience", and "Can you help schedule an interview?" The goal was to ensure that the assistant did not only sound fluent, but remained coherent under follow-up and kept the scheduling path intact.

3. Chat Groundedness

  • Hallucination rate: counted answers that introduced unsupported employers, dates, achievements, project claims, or technical details.
  • Retrieval quality: checked whether the top retrieved chunks matched the real intent of role-fit, RAG, and GitHub project prompts.
  • Answer faithfulness: checked whether the final answer stayed aligned with retrieved evidence instead of embellishing it.

This mattered especially for resume-style prompts, because a plausible but unsupported answer is more harmful than a short honest answer. I therefore treated "sounds good" and "is grounded" as two separate evaluation axes.

4. LLM-as-a-Judge Methodology

I used LLM-as-a-judge as a structured evaluator over (question, retrieved evidence, answer). The judge was asked to score groundedness, retrieval relevance, helpfulness, and hallucination risk. This did not replace manual review, but it made the evaluation more repeatable and easier to reason about across multiple prompts. 

{
  "groundedness_score": 1,
  "retrieval_relevance_score": 1,
  "helpfulness_score": 1,
  "hallucination": false,
  "notes": ""
}

Judge questions:

  • Is the answer fully supported by the retrieved evidence?
  • Are the retrieved chunks relevant to the actual question?
  • Does the answer stay specific without inventing facts?
  • Does the answer remain useful under a follow-up?

5. Failure Modes Found And Fixed

Failure Mode 1: Booking looked complete before it was actually reliable

Google Calendar and Cal.com integrations were inconsistent across local and deployed environments. That created a high-risk failure mode where the product looked booking-ready while the provider layer could still fail.

Fix: moved production booking to a stable Calendly handoff so every visible booking action resolves to a live scheduling flow with confirmation.

Failure Mode 2: Voice scheduling responses sounded unnatural

Early voice tests showed the assistant trying to read a raw URL aloud. Technically correct, but poor for a live conversational experience.

Fix: rewrote the voice prompt to use a spoken booking path and shorter scheduling language.

Failure Mode 3: Public UI exposed weak or low-signal states

The earlier UI surfaced empty areas, technical-looking panels, and noisy source metadata that distracted from the actual demo flow.

Fix: simplified the homepage to the three core actions: ask, book, and call, while keeping the grounding layer intact underneath.

6. What I Would Improve With 2 More Weeks

  1. Add true end-to-end voice booking with tool-driven calendar actions instead of a booking-link handoff.
  2. Build a batch eval harness that runs saved prompts through retrieval, generation, and LLM-as-a-judge scoring to produce reproducible regression metrics.
  3. Expand GitHub indexing so repo-specific answers surface stronger stack, purpose, and tradeoff detail directly from project artifacts.
